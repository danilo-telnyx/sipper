# RFC3261 Compliance Guide

This document details SIPPER's compliance with RFC3261 (SIP: Session Initiation Protocol).

## Overview

SIPPER implements core RFC3261 requirements for SIP message construction, parsing, and validation.

## Implemented Components

### 1. Message Structure (RFC3261 §7)

#### Request Line
```
Method SP Request-URI SP SIP-Version CRLF
```

**Supported Methods:**
- INVITE
- ACK
- OPTIONS
- BYE
- CANCEL
- REGISTER

**Example:**
```
INVITE sip:bob@example.com SIP/2.0
```

#### Status Line
```
SIP-Version SP Status-Code SP Reason-Phrase CRLF
```

**Example:**
```
SIP/2.0 200 OK
```

### 2. Headers (RFC3261 §20)

#### Required Headers (All Messages)

| Header   | Description                          | Format                                    |
|----------|--------------------------------------|-------------------------------------------|
| Via      | Transport path                       | `SIP/2.0/UDP ip:port;branch=z9hG4bK...` |
| From     | Originator                           | `<sip:user@domain>;tag=xyz`              |
| To       | Recipient                            | `<sip:user@domain>;tag=abc`              |
| Call-ID  | Unique call identifier               | `uuid@host`                              |
| CSeq     | Command sequence                     | `123 INVITE`                             |

#### Request-Specific Headers

**INVITE:**
- Contact
- Content-Type (if SDP present)
- Content-Length
- Max-Forwards

**REGISTER:**
- Contact
- Expires
- Max-Forwards

**ACK/BYE:**
- Max-Forwards

**OPTIONS:**
- Accept
- Max-Forwards

### 3. Via Header (RFC3261 §20.42)

**Format:**
```
Via: SIP/2.0/UDP 192.168.1.100:5060;branch=z9hG4bK776asdhds
```

**Magic Cookie:**
- Branch parameter MUST start with `z9hG4bK` (RFC3261 magic cookie)
- SIPPER validates this in compliance checks

**Example:**
```javascript
generateBranch() {
  return `z9hG4bK-${crypto.randomBytes(16).toString('hex')}`;
}
```

### 4. From/To Headers (RFC3261 §20.20, §20.39)

**Format:**
```
From: "Display Name" <sip:user@domain>;tag=1928301774
To: <sip:user@domain>;tag=xyz
```

**Tag Parameter:**
- From tag: MUST be present in requests (generated by UAC)
- To tag: Added in responses by UAS
- Tags identify dialog endpoints

**SIPPER Implementation:**
```javascript
generateTag() {
  return crypto.randomBytes(8).toString('hex');
}
```

### 5. Call-ID (RFC3261 §20.8)

**Format:**
```
Call-ID: f81d4fae-7dec-11d0-a765-00a0c91e6bf6@example.com
```

**Requirements:**
- MUST be globally unique
- MUST remain constant for entire dialog
- Recommended: UUID@host

**SIPPER Implementation:**
```javascript
generateCallId() {
  return `${uuidv4()}@${this.localIP}`;
}
```

### 6. CSeq (RFC3261 §20.16)

**Format:**
```
CSeq: 314159 INVITE
```

**Requirements:**
- Sequence number MUST increment within dialog
- Method MUST match request method
- CANCEL and ACK use same CSeq as INVITE they reference

**SIPPER Validation:**
```javascript
if (parsed.cseq.method !== parsed.method) {
  violations.push('CSeq method must match request method');
}
```

### 7. Max-Forwards (RFC3261 §20.22)

**Format:**
```
Max-Forwards: 70
```

**Requirements:**
- MUST be present in all requests
- Initial value typically 70
- Decremented by each proxy
- Request rejected if reaches 0

**SIPPER Validation:**
```javascript
const maxFwd = parseInt(headers['Max-Forwards']);
if (isNaN(maxFwd) || maxFwd < 0 || maxFwd > 255) {
  violations.push('Max-Forwards must be 0-255');
}
```

### 8. Content-Length (RFC3261 §20.14)

**Format:**
```
Content-Length: 142
```

**Requirements:**
- MUST be present in all messages
- MUST accurately reflect body size in octets
- 0 if no body present

**SIPPER Validation:**
```javascript
const actualLength = Buffer.byteLength(body);
if (actualLength !== contentLength) {
  violations.push('Content-Length does not match body length');
}
```

### 9. SDP (RFC4566)

**Example:**
```
v=0
o=alice 2890844526 2890844526 IN IP4 192.168.1.100
s=Call Session
c=IN IP4 192.168.1.100
t=0 0
m=audio 49170 RTP/AVP 0
a=rtpmap:0 PCMU/8000
```

**SIPPER SDP Builder:**
- Session version (v=)
- Origin (o=)
- Session name (s=)
- Connection (c=)
- Time (t=)
- Media (m=)
- Attributes (a=)

## Authentication (RFC3261 §22)

### Digest Authentication (RFC2617)

**Challenge Format:**
```
WWW-Authenticate: Digest 
  realm="sip.example.com",
  nonce="dcd98b7102dd2f0e8b11d0f600bfb0c093",
  algorithm=MD5,
  qop="auth"
```

**Response Calculation:**
```
HA1 = MD5(username:realm:password)
HA2 = MD5(method:digestURI)
response = MD5(HA1:nonce:nc:cnonce:qop:HA2)
```

**Authorization Header:**
```
Authorization: Digest username="alice",
  realm="sip.example.com",
  nonce="dcd98b7102dd2f0e8b11d0f600bfb0c093",
  uri="sip:sip.example.com",
  response="6629fae49393a05397450978507c4ef1",
  algorithm=MD5,
  qop=auth,
  nc=00000001,
  cnonce="0a4f113b"
```

**SIPPER Implementation:**
```javascript
buildAuthorizationHeader(params) {
  const ha1 = crypto.createHash('md5')
    .update(`${username}:${realm}:${password}`)
    .digest('hex');
    
  const ha2 = crypto.createHash('md5')
    .update(`${method}:${uri}`)
    .digest('hex');
    
  const response = crypto.createHash('md5')
    .update(`${ha1}:${nonce}:${nc}:${cnonce}:${qop}:${ha2}`)
    .digest('hex');
    
  return `Digest username="${username}", realm="${realm}", ...`;
}
```

## Transaction Matching (RFC3261 §17)

### Client Transaction

**Matching Rules:**
1. Same Call-ID
2. Same From tag
3. Same CSeq number
4. Same method (except ACK/CANCEL)
5. Same branch parameter

**SIPPER Note:**
Current implementation matches on transaction timing (first pending transaction receives response). Production implementation should use full matching.

### Server Transaction

Server transactions identified by:
- Via branch parameter
- CSeq method
- Request-URI (for CANCEL)

## Dialog (RFC3261 §12)

### Dialog Establishment

**UAC Dialog State:**
- Call-ID
- Local tag (From tag)
- Remote tag (To tag from response)
- Local sequence number
- Remote sequence number
- Local URI
- Remote URI
- Remote target (Contact)
- Route set

**Example Flow:**
```
INVITE (From tag=abc, Call-ID=xyz)
  → 200 OK (To tag=def)
    → Dialog created: Call-ID=xyz, local tag=abc, remote tag=def
```

### Dialog Termination

Dialog ends when:
- BYE sent/received
- Non-2xx final response to INVITE
- Timeout/error

## Response Codes (RFC3261 §21)

### 1xx - Provisional
- 100 Trying
- 180 Ringing
- 183 Session Progress

### 2xx - Success
- 200 OK

### 3xx - Redirection
- 301 Moved Permanently
- 302 Moved Temporarily

### 4xx - Client Error
- 400 Bad Request
- 401 Unauthorized
- 403 Forbidden
- 404 Not Found
- 407 Proxy Authentication Required
- 408 Request Timeout
- 480 Temporarily Unavailable
- 486 Busy Here
- 487 Request Terminated

### 5xx - Server Error
- 500 Server Internal Error
- 503 Service Unavailable

### 6xx - Global Failure
- 600 Busy Everywhere
- 603 Decline

## SIPPER Compliance Validation

### Automatic Checks

```javascript
SIPParser.validateRFC3261(parsed)
```

**Checks Performed:**
1. Via branch magic cookie present
2. CSeq method matches request method
3. Max-Forwards in valid range
4. Content-Length matches body
5. All required headers present

**Example Output:**
```javascript
{
  compliant: false,
  violations: [
    'RFC3261 violation: Via branch must start with magic cookie z9hG4bK',
    'RFC3261 violation: Content-Length (0) does not match body length (142)'
  ]
}
```

## Method-Specific Requirements

### INVITE (RFC3261 §13)

**Must Include:**
- Via, From, To, Call-ID, CSeq, Max-Forwards
- Contact
- Content-Type and Content-Length if body present

**Flow:**
```
UAC                      UAS
 |                        |
 |---> INVITE            |
 |<--- 100 Trying        |
 |<--- 180 Ringing       |
 |<--- 200 OK            |
 |---> ACK              |
 |                        |
```

### ACK (RFC3261 §13.2.2.4)

**Special Properties:**
- Non-transactional (no response expected)
- Same CSeq number as INVITE
- Method is "ACK"
- Must include To tag from 200 OK

### BYE (RFC3261 §15)

**Requirements:**
- Sent within established dialog
- Uses dialog's Call-ID and tags
- New CSeq number

### REGISTER (RFC3261 §10)

**Requirements:**
- Request-URI is domain (not user)
- Contact header with user's location
- Expires header or expires param
- May require authentication

### OPTIONS (RFC3261 §11)

**Purpose:**
- Query capabilities
- Keepalive
- Network testing

**Requirements:**
- May be sent outside dialog
- No authentication typically required
- Response includes Allow and Accept headers

### CANCEL (RFC3261 §9)

**Requirements:**
- Only cancels pending INVITE
- Same Call-ID, CSeq number, and branch as INVITE
- Method is "CANCEL" (not INVITE)

## Known Limitations

### Not Yet Implemented

1. **TCP/TLS Transport**
   - Currently UDP only
   - TCP/TLS planned for future release

2. **Full Transaction Matching**
   - Current: matches first pending transaction
   - Needed: match by Call-ID + branch + CSeq

3. **Re-INVITE**
   - Session modification not yet implemented

4. **SUBSCRIBE/NOTIFY**
   - Event framework not implemented

5. **MESSAGE**
   - Instant messaging not implemented

6. **REFER**
   - Call transfer not implemented

7. **UPDATE**
   - Session parameter update not implemented

8. **Proxy Behavior**
   - SIPPER is UAC/UAS only, not a proxy

9. **Record-Route/Route**
   - Route set handling simplified

10. **100rel (PRACK)**
    - Reliable provisional responses not implemented
